# 🚫 .gitignore — Complete Notes

> **Level:** Beginner | **Topic:** Telling Git which files to never track

---

## 1. What is .gitignore?

`.gitignore` is a plain text file in your project root where you list all files and folders Git should **completely ignore** — never track, never suggest committing, never show in `git status`.

### 🧒 Simple Analogy
> Imagine you're packing a suitcase (your Git commit). You have a sticky note on the suitcase that says "DON'T pack: dirty laundry, toothbrush, passport." That sticky note is your `.gitignore` — Git will never touch those listed items!

## 4. Creating Your .gitignore

Create a file named **exactly** `.gitignore` in the root of your project. Each line is one pattern:

```
my-project/
├── index.html
├── style.css
├── app.js
├── .gitignore    ← create this file here
├── .env          ← will be ignored
└── node_modules/ ← will be ignored
```

```bash
# Create .gitignore (Mac/Linux)
touch .gitignore

# Open and edit it
code .gitignore     # VS Code
nano .gitignore     # terminal
```

---

## 5. Pattern Syntax — How to Write Rules

```
  PATTERN              WHAT IT IGNORES
  ───────              ───────────────

  secrets.txt          Exact file named "secrets.txt"

  build/               Entire folder named "build" (trailing / = folder)

  *.log                ANY file ending in .log
                       (error.log, debug.log, anything.log)

  **/.env              .env file in the root AND any subfolder
                       (src/.env, config/.env, etc.)

  doc/**/*.pdf         All .pdf files anywhere inside the doc/ folder

  temp-*               Any file starting with "temp-"
                       (temp-backup.js, temp-notes.txt, etc.)

  !important.log       EXCEPTION — do NOT ignore this specific file
                       (overrides the *.log rule above)

  # This is a comment  Lines starting with # are comments, not rules
```

## 8. The "Already Tracked" Trap ⚠️

This is the most common beginner mistake:

```
  TIMELINE:                               PROBLEM:
  ─────────                               ────────

  Day 1:  git add .env                    .env is now TRACKED by Git!
          git commit "first commit"

  Day 5:  You add .env to .gitignore      TOO LATE — .gitignore only
                                          affects UNTRACKED files.
                                          Git keeps tracking .env anyway!

  git status still shows .env changes 😱
```

### The Fix:

```bash
# Step 1: Tell Git to stop tracking the file
#         (it stays on YOUR disk, just removed from Git's memory)
git rm --cached .env

# For a whole folder:
git rm -r --cached node_modules/

# Step 2: Make sure .gitignore has the rule
echo ".env" >> .gitignore

# Step 3: Commit the removal
git commit -m "Stop tracking .env — add to .gitignore"

# Step 4: Push to GitHub (now .env is gone from GitHub)
git push
```

```
  AFTER git rm --cached .env:

  Your computer:    .env still exists ✅ (you can still use it)
  Git history:      .env is no longer tracked ✅
  GitHub:           .env removed from future commits ✅

  ⚠️  BUT: Old commits still have .env in history!
           If it was a real password → CHANGE IT IMMEDIATELY
           on the service where you use it!
```

## 9. Check If a File Is Being Ignored

```bash
# Check if a specific file is ignored
git check-ignore -v .env

# Output if ignored:
.gitignore:3:.env   .env
#           ▲  ▲
#           │  └── the pattern that matched
#           └── line number in .gitignore

# No output = file is NOT ignored
```

## 12. Interview Questions

---

### Q1: What is `.gitignore` and why is it important?
**A:** `.gitignore` is a file that tells Git which files and folders to never track. It's important for three reasons: **security** (keeping secrets like `.env` files out of GitHub), **size** (excluding huge generated folders like `node_modules`), and **cleanliness** (excluding OS junk like `.DS_Store` that pollutes the repo for teammates).

---

### Q2: I added `.env` to `.gitignore` but `git status` still shows it. Why?
**A:** Because `.env` was already committed to Git before you added it to `.gitignore`. Once a file is tracked, `.gitignore` has no effect on it. The fix is: `git rm --cached .env` to untrack it, then commit that change. After that, `.gitignore` will work correctly.

---

### Q3: What is the difference between `.gitignore` and `git rm --cached`?
**A:** `.gitignore` tells Git to **ignore a file going forward** — it only works on untracked files. `git rm --cached` tells Git to **stop tracking a file it was already tracking**, without deleting it from your local disk. You often need both: `rm --cached` to stop tracking, then `.gitignore` to prevent re-tracking.

---

### Q4: How do you ignore all `.log` files but keep one specific log file?
**A:** Use the `!` exception syntax: first write `*.log` to ignore all logs, then on the next line write `!important.log` to make an exception for that specific file. Git processes rules top-to-bottom, so the exception must come after the general rule.

---

### Q5: Should you commit the `.gitignore` file itself to Git?
**A:** Yes, absolutely. The `.gitignore` file should always be committed so all teammates automatically get the same ignore rules. It's project configuration — not a secret — and it ensures the whole team ignores the same files (like `node_modules/` and `.env`).

---

### Q6: What happens to ignored files in old commits?
**A:** They remain in the old commits forever. `.gitignore` only affects future commits — it does not rewrite history. If you accidentally committed a secret, you need special tools like `git filter-branch` or BFG Repo Cleaner to purge it from history. It's far easier to prevent this by setting up `.gitignore` on day one.

---

### Q7: What is `git check-ignore` used for?
**A:** `git check-ignore -v filename` tells you whether a file is being ignored and which rule in `.gitignore` is causing it. Very useful for debugging when you expect a file to be ignored but it isn't (or vice versa).
